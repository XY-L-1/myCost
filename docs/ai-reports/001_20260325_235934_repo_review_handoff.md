# Project Repository Review

## 1. Project Overview
- This app appears to be a mobile personal expense tracker with optional account creation, guest/offline usage, local-first data entry, category management, and Supabase-backed sync for signed-in users.
- The target user appears to be an individual tracking day-to-day spending rather than a full bookkeeping/business user. The implemented flows focus on expense capture, monthly review, and simple category breakdowns rather than ledgers, invoices, or reconciliation.
- Current product maturity: early-to-mid MVP. The app has real end-user flows and a meaningful local/sync architecture, but the feature surface is still narrow and several core product and architectural risks remain unresolved.
- Confidence level of analysis: Medium-High. Confidence is high for the mobile client because the repository is small and readable. Confidence is lower for backend guarantees because the Supabase schema, RLS policies, and deployment configuration are not present in the repo.

## 2. Tech Stack
- Frameworks/libraries: Expo 54, React 19, React Native 0.81, React Navigation 7, Zustand, Zod, Supabase JS, Expo SQLite, Expo Secure Store, React Native SVG, community DateTimePicker.
- Language(s): TypeScript throughout app code; generated native iOS/Android code present.
- Storage/database: Local SQLite via `expo-sqlite`; remote sync target is Supabase tables named `expenses` and `categories`.
- State management: Zustand for app init/auth; React local state for screen data; two lightweight React contexts for auth-gate and sync-gate values.
- Navigation/routing: React Navigation native stack plus bottom tabs; auth stack and main app stack separated.
- Charting / UI libraries: `react-native-svg` for a custom donut chart; otherwise mostly hand-built React Native UI.
- Testing tools: No app test runner or test config is present. No Jest/Vitest/Detox/Maestro setup was found in repo-level config.
- Build/deployment-related tools: Expo CLI scripts only (`start`, `android`, `ios`, `web`); native `ios/` and `android/` projects are committed; no EAS config, Fastlane, or CI workflow was found.
- Inferred or uncertain items:
  - The remote Supabase database likely has at least `expenses` and `categories` tables and probably some uniqueness rule around normalized category names, because the sync code handles `23505` conflicts. That constraint is inferred from client behavior, not proven by repo schema.
  - The app is intended to be offline-capable, but user-facing offline state and conflict UX are minimal.

## 4. Architecture Summary
- Architectural style used:
  - Functional React Native screens on top of a thin repository layer.
  - Local-first persistence in SQLite.
  - Sync layer that pushes dirty records and pulls remote data from Supabase.
  - App-level orchestration concentrated in `App.tsx`.
- Data flow:
  - UI screens call repository methods directly for reads/writes.
  - Repositories write to SQLite and emit mutation events via `[syncEvents.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/sync/syncEvents.ts)`.
  - `App.tsx` subscribes to those mutation events and triggers push sync for authenticated users.
  - On login, `App.tsx` runs a larger sync pipeline: attach anonymous data, pull categories, repair duplicate categories, seed defaults, pull expenses, repair missing category references, then push dirty categories and expenses.
- UI layer organization:
  - Each screen owns its own styles, view logic, data fetching, and loading state.
  - There is no shared design system, no reusable `src/components` usage, and several helper functions such as currency/date formatting are duplicated across screens.
- Business logic placement:
  - Some domain logic is in service files, which is good.
  - A large amount of lifecycle and sync logic lives in `App.tsx`, which makes bootstrap behavior harder to reason about and test.
  - Screen components still contain business-adjacent logic such as filtering, category creation rules, and some date formatting behavior.
- Data/storage/service layers:
  - `[database.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/db/database.ts)` is a minimal SQLite adapter.
  - `[migration.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/db/migration.ts)` defines the local schema.
  - `[expenseRepository.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/repositories/expenseRepository.ts)` and `[categoryRepository.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/repositories/categoryRepository.ts)` are the local data access layer.
  - `[syncService.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/sync/syncService.ts)` is the remote sync layer.
- Reusability and modularity assessment:
  - Moderate at the storage/service level.
  - Weak at the UI and app-orchestration level.
  - Layering is conceptually present but not consistently enforced. For example, repositories claim to avoid sync concerns but still emit sync events.

## 7. Data Model and Persistence
- Core entities found in code:
  - `Expense`: `id`, `amountCents`, `currency`, `categoryId`, `description`, `expenseDate`, `createdAt`, `updatedAt`, `deletedAt`, `dirty`, `version`, `deviceId`, `userId`.
  - `Category`: `id`, `name`, `normalizedName`, `createdAt`, `updatedAt`, `deletedAt`, `dirty`, `version`, `deviceId`, `userId`.
- How data is stored/retrieved:
  - Local storage is SQLite via `[database.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/db/database.ts)`.
  - Local schema is created in `[migration.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/db/migration.ts)`.
  - Expenses and categories are read and written through repository classes.
  - Signed-in sync uses Supabase tables via `[syncService.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/sync/syncService.ts)`.
  - Auth session and device ID are stored in Secure Store via `[supabaseClient.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/auth/supabaseClient.ts)` and `[appInitStore.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/state/appInitStore.ts)`.
- Schema/model design quality:
  - Stronger than a typical MVP for sync metadata. `dirty`, `version`, `deviceId`, `userId`, and tombstones show intentional offline-sync design.
  - Weak for broader product growth. There are no tables for budgets, accounts, recurring rules, settings, income, attachments, or audit trails.
  - `currency` exists, which is forward-looking, but the rest of the app behaves as single-currency USD-only.
- Potential data consistency risks:
  - Local read queries do not scope by `userId`, so data for different users on the same device can mix in the UI. Evidence: `[categoryRepository.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/repositories/categoryRepository.ts)` and `[expenseRepository.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/repositories/expenseRepository.ts)` query all non-deleted rows by month/name without a user filter.
  - `signOut` explicitly keeps local SQLite data, which is dangerous combined with unscoped queries. Evidence: `[authStore.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/auth/authStore.ts)`.
  - SQLite schema does not declare foreign keys between `expenses.categoryId` and `categories.id`.
  - Delete operations do not update `updatedAt` or increment `version`, which weakens sync conflict metadata. Evidence: `[expenseRepository.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/repositories/expenseRepository.ts)`.
  - Dates are stored as text and filtered with `LIKE 'YYYY-MM%'`, which is simple but brittle.
  - `sync_metadata` is created but currently unused.
- Migration/versioning concerns:
  - Local migrations are incremental and guarded by a `migrations` table.
  - There are no transaction wrappers around each migration batch.
  - There are no down migrations or schema health checks.
  - Remote schema migration/versioning is not represented in the repo, so compatibility expectations between app versions and backend versions are unclear.

## 10. Performance and Scalability
- Potential bottlenecks:
  - Expense sync pushes records one by one rather than batching.
  - Category sync also processes rows serially and may perform extra lookups per row.
  - Expense list preloads 12 months, then grows with additional 6-month fetches and filters in JS memory.
- Expensive rendering or state patterns:
  - Most screens are small and likely fine today.
  - The bigger issue is repeated data fetching on both focus and sync-status changes, which can cause redundant reloads.
- Storage/query inefficiencies:
  - Expense queries use string prefix filtering on `expenseDate` instead of stronger date indexing or range queries.
  - Category and expense reads do not filter by `userId`, which is both a privacy bug and a scalability problem as local data accumulates.
  - Summary trend requires multiple separate monthly total queries.
- Scalability concerns as data grows:
  - History screens will become slower as more months are loaded and filtered client-side.
  - Cross-device expense freshness is weak because expenses are not periodically pulled after login.
  - There is no pagination at the record level, only month-batch loading.
- Maintainability concerns as features expand:
  - Adding budgets, recurring entries, or accounts on top of the current `App.tsx` orchestration and screen-owned logic will increase coupling quickly.
  - Repeated visual/system code will become harder to evolve consistently.

## 11. Security / Privacy Review
- Handling of financial/user data:
  - Financial data is stored locally in SQLite. There is no evidence of database encryption or an app lock/biometric gate protecting the finance data itself.
  - Secure Store is used for auth persistence and device ID, which is appropriate for those values.
- Secrets/config risks:
  - `[.env](/Users/aic/Desktop/mycostp/expense-tracker/.env)` is committed to the repo. The included key is a Supabase anon key, which is not a server secret, but committing environment files is still weak configuration hygiene.
  - Backend security posture is uncertain because Supabase RLS policies are not in repo.
- Authentication/session issues:
  - Auth store initializes from `getSession()` but does not subscribe to `onAuthStateChange`, so auth state changes outside the direct sign-in/sign-out flow may not fully propagate through the store.
  - Sign-up verification UX is manual and could confuse users if session state changes asynchronously.
- Local storage sensitivity:
  - The largest issue in this codebase is local data isolation. Because local expense/category queries are not user-scoped and sign-out does not clear local DB state, one user can potentially see another user’s financial data on the same device.
  - Android manifest enables backups and includes storage permissions that do not appear necessary for the implemented feature set. Evidence: `[AndroidManifest.xml](/Users/aic/Desktop/mycostp/expense-tracker/android/app/src/main/AndroidManifest.xml)`.
- Input sanitization / validation issues:
  - Form validation is minimal.
  - Expense edit accepts free-form date text.
  - Category duplicate detection in the add flow is weaker than the normalization logic used elsewhere.
- Any risky assumptions:
  - The app assumes local data can safely persist across auth state changes.
  - The app assumes backend conflict resolution and RLS are correctly configured, but that is not verifiable from this repository.

## 12. Technical Debt / Risk Areas
- Title: User data isolation is broken locally
  - Severity: High
  - Why it matters: A signed-out user, a guest user, or a second signed-in user on the same device can potentially see the first user’s expenses/categories. For a finance app, this is a major privacy and trust issue.
  - Evidence: [expenseRepository.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/repositories/expenseRepository.ts), [categoryRepository.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/repositories/categoryRepository.ts), [authStore.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/auth/authStore.ts)
  - Suggested direction: Make all local reads explicitly user-scoped, define a safe guest scope, and decide whether sign-out should clear or archive local user partitions.
- Title: App bootstrap and sync orchestration are concentrated in `App.tsx`
  - Severity: High
  - Why it matters: Startup, auth, login merge, retry logic, mutation sync, polling, and navigation reset are all mixed together, making bugs harder to isolate and new features harder to add.
  - Evidence: [App.tsx](/Users/aic/Desktop/mycostp/expense-tracker/App.tsx)
  - Suggested direction: Extract a dedicated sync coordinator and app bootstrap/use-case layer with smaller tested units.
- Title: Delete metadata is too weak for robust sync conflict handling
  - Severity: High
  - Why it matters: Deletes should be first-class state transitions. Not updating `updatedAt` and `version` on delete can produce conflict ambiguity and stale remote/local resolution.
  - Evidence: [expenseRepository.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/repositories/expenseRepository.ts), [syncService.ts](/Users/aic/Desktop/mycostp/expense-tracker/src/sync/syncService.ts)
  - Suggested direction: Treat delete like update: bump `updatedAt`, increment `version`, and test conflict cases explicitly.
- Title: Expense sync freshness is incomplete after login
  - Severity: Medium
  - Why it matters: On a multi-device account, remote expense changes may not appear until the next login cycle because only categories get background pull refresh.
  - Evidence: [App.tsx](/Users/aic/Desktop/mycostp/expense-tracker/App.tsx)
  - Suggested direction: Add periodic or event-driven expense pull, plus visible sync status.
- Title: Date and currency handling are inconsistent
  - Severity: Medium

## 15. AI Handoff Summary
- This repository is a React Native/Expo personal expense tracker that stores data in local SQLite and optionally syncs signed-in user data to Supabase. The implemented product is centered on expense CRUD, categories, a home dashboard, and a monthly summary view.
- The strongest technical idea in the repo is the offline-first sync model: expenses and categories carry `dirty`, `version`, `deviceId`, `userId`, and tombstone metadata, and the app repairs/merges category data on login. The weakest area is data isolation and client architecture: local reads are not user-scoped, sign-out keeps local data, sync/bootstrap logic is heavily concentrated in `App.tsx`, and there are no tests.
- What is working: guest mode, auth, local expense entry, edit/delete, local persistence, category creation/seeding, monthly reporting, and Supabase push/pull basics.
- What is weak: privacy boundaries, sync completeness, delete semantics, date/currency consistency, error visibility, test coverage, and missing bookkeeping features such as budgets and recurring transactions.
- What should be improved first: fix user-data partitioning, repair sync correctness around deletes and refresh, add tests, then refactor bootstrap/sync architecture before expanding product scope.
